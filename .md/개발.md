## 1. 아키텍처 개요

```
┌─────────────────────────────────────────────────────────────┐
│                        Clients                              │
│  ┌──────────────────┐       ┌──────────────────┐           │
│  │  apps/web        │       │  apps/mobile     │           │
│  │  Next.js (PWA)   │       │  Expo (RN)       │           │
│  │  SEO · AEO       │       │  IAP · Push · Ad │           │
│  └────────┬─────────┘       └────────┬─────────┘           │
│           │                          │                      │
│           └──────────┬───────────────┘                      │
│                      ▼                                      │
│           packages/api-client (orval 생성)                  │
│           packages/schemas   (zod — 공유 검증)              │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  apps/web — Next.js Route Handlers (BFF / API)              │
│  zod 검증 · Prisma · Supabase Auth 토큰 검증                │
└──────────────┬──────────────────────────────┬───────────────┘
               ▼                              ▼
     Supabase Postgres              AWS S3 (이미지)
     (Prisma ORM)                   presigned URL 업로드
               │
     Supabase Auth (Email/Password)
```

**핵심 원칙**

- **웹·앱 UI는 분리**, **API·타입·스키마는 공유** (monorepo)
- DB 접근은 **서버(Route Handler)에서만** — 클라이언트 → Supabase Postgres 직접 접근 없음
- 인증은 **Supabase Auth**, 데이터는 **Prisma**
- API 클라이언트는 **orval만** 사용 (타입 + TanStack Query hooks 일괄 생성)
- 웹 UI는 **shadcn/ui** + [DESIGN.md](./DESIGN.md) 토큰
- 모든 타입·스키마는 **SSOT** — 원본 하나에서만 정의하고 `Pick`·`Omit`·`extends`로 파생 ([§2.7](#27-개발-원칙--ssot-single-source-of-truth))

### 2.7 개발 원칙 — SSOT (Single Source of Truth)

참고: [SSOT(Single Source of Truth) 패턴](https://blog.story-dict.com/posts/tli/03-ssot)

같은 데이터를 타입·스키마로 **두 번 정의하지 않는다**. 원본은 **한 곳**에만 두고, 화면·API·폼 타입은 `Pick`·`Omit`·`extends`·`Partial`로 **파생**한다.

**StoryEcho SSOT 계층**

| 계층 | SSOT (원본) | 파생 · 금지 |
| --- | --- | --- |
| **DB** | `prisma/schema.prisma` → `@prisma/client` | Route Handler에서 Prisma 타입 import. `interface DbUser { ... }` 중복 금지 |
| **API 계약** | `packages/schemas` (zod) → `openapi.json` → **orval** (`packages/api-client`) | orval generated 파일 **직접 수정 금지**. 화면용 타입은 generated에서 `Pick`/`Omit` |
| **런타임 검증** | zod (`packages/schemas`) — OpenAPI와 **동일 스키마** | API용 zod와 “프론트만의 zod” 이중 정의 금지 |
| **디자인 토큰** | [DESIGN.md](./DESIGN.md) → `apps/web/app/globals.css` CSS 변수 | hex 하드코딩·컴포넌트마다 색상 재정의 금지 |
| **UI 컴포넌트 (web)** | shadcn `components/ui/*` + DESIGN.md 변수 | 페이지마다 raw `<button>` 스타일 복붙 금지 — shadcn variant 사용 |

**타입 파생 예시 (StoryEcho)**

```ts
// packages/api-client — orval generated (수정 ❌)
import type { Story } from '@storyecho/api-client';

// apps/web/features/stories/types.ts — 파생만 (수정 ✅)
export interface StoryListItem extends Pick<Story, 'id' | 'bodyText' | 'createdAt'> {}

export interface StoryFormInput extends Pick<Story, 'bodyText'> {
  visibility: 'private' | 'community';
}

export interface CreateStoryPayload extends Omit<StoryFormInput, never> {}
```

```ts
// ❌ 화면마다 Story 재정의
interface StoryCard { id: string; bodyText: string; }

// ✅ orval Story에서 파생
interface StoryCard extends Pick<Story, 'id' | 'bodyText'> {}
```

**개발 시 체크리스트**

1. 새 `interface` 작성 전 — 원본(Prisma / orval / zod)에서 `Pick`·`Omit`·`extends`로 가능한지 확인
2. 파생 타입은 `features/<도메인>/types.ts` 등 **원본 옆 레이어**에만 둠
3. `packages/api-client/src/generated/**` — generate 결과물, **커밋·수정하지 않고** CI/`pnpm generate:api`로 재생성
4. zod 스키마 변경 → OpenAPI → orval **한 번에** 갱신 (API 계약 SSOT 유지)
5. shadcn 컴포넌트 추가는 CLI로만 — 스타일은 CSS 변수(DESIGN.md)로 통일

### 5.1 Vercel Cron (추후 도입)

[Cron Jobs 요금·제한](https://vercel.com/docs/cron-jobs/usage-and-pricing) 기준:

| 플랜 | Cron 포함 | 실행 빈도 | 시각 정확도 |
| --- | --- | --- | --- |
| **Hobby (Free)** | ✅ 포함 | **하루 1회**만 가능 | ±59분 (예: 20:00 설정 → 20:00~20:59) |
| **Pro ($20/월)** | ✅ 포함 | **최소 1분** 간격 | 분 단위 |
| Enterprise | ✅ 포함 | 최소 1분 | 분 단위 |

**Pro가 필요한 경우**

- cron을 **하루 2번 이상** 돌려야 할 때 (Hobby는 배포 자체가 실패)
- **분·시간 단위** 정확한 스케줄 (예: 매시간 타임존별 20:00 알림)
- StoryEcho MVP(매일 질문 1회 + 일일 배치)만이면 **Hobby로 1개 cron**으로 시작 가능

**StoryEcho cron 후보 (도입 시 1 job으로 묶기 권장 — Hobby 1일 1회 제한)**

```
매일 1회 실행 → 한 Route Handler에서 순서 처리:
  1. 오늘의 질문 배정
  2. Echo 해당일 사용자 처리
  3. 타임캡슐 unlock_at 도래 처리
  4. (푸시) 알림 ON 사용자 대상 발송 — 20:00 ±59분 허용 또는 Pro 업그레이드
```

**설정 방법 (도입 시)**

1. `vercel.json`에 cron 등록:

```json
{
  "crons": [
    {
      "path": "/api/cron/daily",
      "schedule": "0 11 * * *"
    }
  ]
}
```

<!-- TODO: 입력하기 -->
> `schedule`은 **UTC** 기준. KST 20:00 ≈ UTC 11:00 (서머타임 시 조정 필요).

2. Route Handler에서 `CRON_SECRET` 검증:

```ts
// apps/web/app/api/cron/daily/route.ts
export async function GET(request: Request) {
  const authHeader = request.headers.get('authorization');
  // TODO: 입력하기 — CRON_SECRET env와 비교
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return new Response('Unauthorized', { status: 401 });
  }
  // ... 배치 로직
}
```

3. Vercel 대시보드 → Project → Settings → Cron Jobs에서 실행 로그 확인

**Hobby 한계 우회 (Pro 없이)**

- 외부 스케줄러(cron-job.org 등)가 `GET /api/cron/daily` + `Authorization: Bearer CRON_SECRET` 호출
- Vercel Cron과 동일 엔드포인트 재사용 가능

## 8. API · 타입 설계 원칙

1. **REST** + JSON, `/api/v1/...` prefix
2. 모든 mutation은 **zod parse** (`packages/schemas`) 후 Prisma — zod가 API SSOT
3. 인증: `Authorization: Bearer <supabase_jwt>` 또는 web cookie session
4. 게스트: `X-Device-Id: <uuid>` + guest JWT 또는 server-side guest row
5. 응답 envelope: `{ data, error }` 또는 HTTP status + `{ message, code }`
6. OpenAPI spec 변경 → `pnpm generate:api` (orval) → CI diff 체크
7. **SSOT**: Request/Response 타입은 zod → orval 파이프라인에서만 정의. 프론트·Route Handler에서 동일 필드를 가진 `interface` 중복 작성 금지
8. UI·화면 타입은 orval/Prisma 원본에서 **`Pick`·`Omit`·`extends`로만** 파생 ([§2.7](#27-개발-원칙--ssot-single-source-of-truth))

### 주의 · 트레이드오프

| 이슈 | 설명 |
| --- | --- |
| **Next.js + Expo 이중 UI** | 화면 16개 × 2 플랫폼 — MVP는 **핵심 화면만 native** |
| **Prisma + Supabase Auth** | Auth user id = Prisma `users.id` FK 동기화 hook 필요 (가입 시 upsert) |
| **SSOT vs shadcn** | shadcn은 컴포넌트 SSOT, 색·radius는 DESIGN.md → CSS 변수 SSOT. variant hex 하드코딩 금지 |

| **S3 public vs presigned GET** | CDN 없이 public bucket이 단순. private bucket이면 조회마다 presigned GET |
| **Vercel Cron Hobby** | 하루 1회·±59분 — 20:00 정각 알림이 중요하면 Pro 또는 외부 스케줄러 |
